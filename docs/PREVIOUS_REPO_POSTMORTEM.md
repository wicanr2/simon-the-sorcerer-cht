# 前一版中文化失敗分析(Postmortem)

分析對象:前一版《神通妙巫師》中文化(deepseek + glm5.2 產出,`~/scummvm/simon-1`,
repo `simon-the-sorcerer-cht-scummvm-opencode`)。

先說公道話:**前一版的翻譯文筆本身不差**,抽樣讀來通順自然。它的失敗**不在中文品質,而在方法學**——
沒有先把「要翻什麼、從哪讀、怎麼畫」對齊引擎的真實模型,就開始翻,結果一路報「完成」,直到玩家一開遊戲才發現破綻。

以下六點,每一點都附可驗證的證據。

<a name="1"></a>
## 1. 沒有真正的「翻譯分母」,ID 空間對錯了

前一版的字串萃取,把**檔案 offset 當成 string ID**。例如 `strings/still_untranslated.txt` 裡用
`0x1010`(= 4112)這類十六進位當「ID」。但 AGOS 引擎的實際模型是:

- GAMEPC 內建字串表:`stringId < 0x8000`,總數由 GAMEPC 檔頭第 4 個 `UInt32BE` 決定 = **368 條**。
- 對白:`stringId >= 0x8000`,按需從 GME 的 TEXTxx 分頁載入。

`4112` 遠超過 368 的索引上限——**那個數字根本不是引擎的 string ID**。整個枚舉從一開始就沒對齊引擎,
於是「總共要翻幾條」這個分母無從得知,覆蓋率也無法驗證。

> 對照本版:以引擎真實 ID 為準,權威分母見 [`strings/scope.md`](../strings/scope.md);全劇 4035 條。

<a name="2"></a>
## 2. 對錯了版本:英文 CD 版天生缺四成字幕

這是最根本、也最沒被診斷出來的問題。前一版以**英文 CD talkie 版**為底做中文化。
但 ScummVM 原始碼註解白紙黑字:

> `// English and German versions don't have full subtitles`  (`agos.cpp`)

Adventure Soft 把 1993 floppy 版加語音做成 CD 版時,為了塞語音,**把大量對白的文字從資料裡拿掉了**。
實測缺口:

| | floppy(完整) | CD(前一版的底) |
|---|---|---|
| 物件字串 | 694 | 368 |
| 對白 | 3342 | ~1461 |

CD 版少了**約四成**文字。玩家看到的「角色嘴巴在動、卻一個字都沒有」就是這個——**不是 bug,是資料先天缺**。
前一版對著一個殘缺的來源翻,再怎麼翻都補不回那四成。

> 對照本版:改以**文字完整的 floppy 版**為字幕來源,融合 CD 語音。見 [`docs/FUSION_DESIGN.md`](FUSION_DESIGN.md)。

<a name="3"></a>
## 3. 注入方式用「英文字串比對」,救不了語音-only 的行

前一版在 `getStringPtrByID` 之後,拿**螢幕上的英文字串**去查表換中文(`translateInPlace`,以英文為 key)。
問題:語音-only 的行**根本沒有英文字串**(script 裡是 `NULL_STRING`)。沒有英文可比對,就沒有東西可替換。

也就是說,即使前一版把字幕強制打開,那些行仍然一片空白——因為它的方法在那些行上**天生無解**。

> 對照本版:改以**行的身分(floppy stringId / speechId)**為 key 注入,CD 沒文字的行也掛得上字幕。

<a name="4"></a>
## 4. 完全漏掉硬編碼 UI —— 操作選單永遠是英文

動詞列(走到 / 查看 / 使用⋯)不是遊戲資料,而是**寫死在 ScummVM 原始碼** `verb.cpp` 的 C 陣列
`english_verb_names[]`,`printVerbOf()` 依 `_language` 選陣列,而且**不經過 `getStringPtrByID`**。

前一版在 `translations/zh.tsv` 裡放了「Walk to→走到」這類譯文,但因為動詞列不走查表那條路,**那些譯文是死的、永遠不會被用到**。結果玩家實測到的操作選單一直是英文(這正是本次接手的起點)。

同理,存讀檔訊息(`saveload.cpp`)也是硬編碼、無繁中分支。

> 對照本版:在 `verb.cpp` / `saveload.cpp` 原始碼加 `ZH_TWN` 分支(Big5),從引擎層中文化。

<a name="5"></a>
## 5. 字太小 + 假字型

- `simon_zh12.dcjk` 與 `simon_zh16.dcjk` **byte 完全相同**,檔頭都是 `w=12 h=12`——宣稱的「16×16 大字對白」是假的,實際只有 12×12。
- 12×12 中文筆畫糊成一團、幾乎不可讀。retro CJK 的正解是**拉大字級(24×24)、不硬塞小位**,前一版反其道而行。

> 對照本版:24×24 真點陣字型;為容納全形字,加大 AGOS 字幕文字緩衝(6400→40000 bytes)。

<a name="6"></a>
## 6. 用「自稱條數」當完成標準

前一版 README 寫「總計 **1237 / ~1237 ✅ 完成**」。但:

- 分母 `~1237` 是**捏造**的(真實約 1850+,含硬編碼 UI 更多)。
- 同一 repo 的 `strings/still_untranslated.txt` 還躺著 **1168 條**未翻。
- WORKLIST 多處標 `[x] 完成`,實際功能(動詞列、字幕完整度)並未達成。

「自稱完成」與「實機可玩」之間有巨大落差。驗收標準錯了:應以**實機看得到中文 + dump 未翻歸零**為準,而非翻譯表的自報條數。

> 對照本版:以實機截圖 + `comm` 比對 all_ids 全覆蓋驗收;證據見 `screenshots/PROOF_*.png`。

<a name="root"></a>
## 7. 根因總結與教訓

六個症狀,一個根因:**沒有在動手前,用引擎原始碼把「中文化的範圍」調查清楚。**

| 教訓 | 具體做法 |
|---|---|
| 先建權威分母,再翻譯 | 用引擎 ID 空間(非檔案 offset)枚舉,列成 `scope.md` |
| 先確認資料完整度 | 診斷「CD 版缺字幕」這種資料層問題,選對來源版本 |
| 注入以「行的身分」為 key | 而非以易失效的英文字串比對 |
| 別漏「不經查表」的硬編碼字 | 動詞列 / 存讀檔在原始碼,需 source patch |
| retro CJK 拉大字級 | 24px,不縮字塞小位 |
| 以 code 與實機為真相 | 不信任自我標記的「完成」,dump 驗證歸零 |

與本版的逐項量化對照見 [`docs/COMPARISON.md`](COMPARISON.md);中文化難度的完整討論見
[`docs/LOCALIZATION_DIFFICULTY.md`](LOCALIZATION_DIFFICULTY.md)。
