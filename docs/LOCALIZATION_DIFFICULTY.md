# 魔法師西蒙中文化的難度

這份文件記錄《Simon the Sorcerer》(魔法師西蒙,1993,Adventure Soft,AGOS 引擎)繁體中文化過程中遇到的實際難點。目的不是抱怨,而是把「為什麼這個看似單純的『把英文換成中文』會這麼麻煩」講清楚,讓後續接手或做其他 AGOS 遊戲的人少踩坑。

一句話總結:**這款遊戲的中文化難點,九成不在翻譯,而在「文字到底藏在哪、怎麼畫出來、以及英文 CD 版根本沒有完整字幕」這三件事上。**

---

## 0. 先講結論:難度等級對照

| 面向 | 一般 SCUMM 中文化(如印第安納瓊斯) | 魔法師西蒙(AGOS) |
|------|-----------------------------------|-------------------|
| 引擎 CJK 基礎 | 內建(`loadCJKFont` / 雙位元組渲染) | **零**,全部從頭建 |
| 文字來源 | 集中在 bytecode | **6 大類、散在 4 個地方** |
| UI 字(動詞/存檔) | 隨資料一起 | **寫死在 C++ 原始碼** |
| 字幕完整度 | 完整 | **英文 CD 版不完整(有語音無字幕)** |
| 取得原始文字 | 直接讀 | **floppy 版被自訂壓縮,要跑 DOS 安裝** |

SCUMM 那條路,補幾十行就能接上 CJK;AGOS 這條,光是「找齊要翻的字」就是一個小型逆向工程專案。

---

## 1. AGOS 引擎完全沒有 CJK 基礎

ScummVM 的 SCUMM 引擎(v7 以上)本來就為日文/韓文/中文設計過:有 `loadCJKFont`、雙位元組字元偵測、`get2byteCharPtr`、`_useCJKMode`。做中文化基本上是「開開關、補張字型表」。

AGOS 引擎**沒有任何這些東西**。它假設每個字元都是單一 byte、用一張硬編碼的 8×6 點陣英文字型(`english_simonFont[]`)畫字。要顯示中文,必須自己補齊:

- Big5 雙位元組的偵測與狀態機(繪字函式一次只收到一個 byte,中文要兩個 byte 組起來)
- 一套點陣 CJK 字型的載入與繪製
- 中文沒有空格,得自己寫「按字寬換行」的邏輯(原版換行是找空格)

**這一項就決定了本專案不是「patch 幾十行」,而是要新增一個模組。**

---

## 2. 文字散在 6 個地方,還分兩套渲染

要「把英文換成中文」,得先回答「這個字從哪來、怎麼畫出來」。西蒙的螢幕文字有六大來源:

| 類別 | 來源 | 能不能用查表攔到 |
|------|------|-----------------|
| 物品名、房間名、短提示 | GAMEPC 內建字串表 | ✅ 走 `getStringPtrByID` |
| 對白、旁白 | 遊戲資料的 TEXT 區塊 | ✅ 走 `getStringPtrByID` |
| 物品欄描述 | 同上 | ✅ |
| **動詞列(走到/查看…)** | **寫死在 `verb.cpp` 的 C 陣列** | ❌ 完全不經查表 |
| **存/讀檔訊息** | **寫死在 `saveload.cpp`** | ❌ |
| 片頭 / 片尾 logo | VGA 預繪點陣圖 | ❌ 得改美術 |

前一版中文化只處理了「走查表那條路」就宣告完成,結果玩家看到的**操作選單永遠是英文**——因為那些字根本不在遊戲資料裡,而是編譯進執行檔的 C 常數。要翻它們,得改 ScummVM 原始碼、加一組繁中陣列。

**教訓:動手翻譯前,得先把「螢幕上每一種字各自從哪讀出來」查清楚,否則一定漏。**

---

## 3. 最硬的一點:英文 CD 版根本沒有完整字幕

這是整個專案最出乎意料、也最關鍵的發現。

實際測試時會看到:**角色嘴巴在動,但畫面一個字都沒有。** ScummVM 的原始碼註解直接點破原因:

> `// English and German versions don't have full subtitles`

《魔法師西蒙》有兩種版本:1993 的 **floppy 版**(純文字、無語音)和後來的 **CD talkie 版**(加了語音)。做 CD 版時,Adventure Soft 為了塞進語音,**把大量對白的文字字串從資料裡拿掉了**——那些行只剩語音,沒有文字。

量化一下缺口(本專案實測):

| | floppy 版(完整) | CD 版(現況) |
|---|---|---|
| 物件字串 | 694 | 368 |
| 對白 | 3342 | ~1461 |

**CD 版少了約四成的文字。** 這代表:無論翻譯表做得多完整,只要以 CD 版為底,那些「有語音無文字」的行永遠是空白——因為根本沒有英文原文可以替換成中文。

這也解釋了為什麼前一版「翻了一千多條卻還是一堆空白」:它對著一個先天殘缺的文字來源在翻。

**解法**是「融合」:以文字完整的 **floppy 版為底**,補上 **CD 版的英語語音**。這樣字幕 100% 完整,語音照舊。但要把兩個版本接起來,又帶出下面幾個新難點。

---

## 4. 完整文字被鎖在自訂壓縮的安裝磁片裡

floppy 版的完整文字不是現成檔案,而是壓在三張安裝磁片的 `GAME.RED` 裡——Adventure Soft 自訂的「RR」安裝壓縮格式,不是 zip、不是常見 LZ,手動逆向不划算。

繞法:**用 DOSBox 在容器裡跑原版 DOS 安裝程式**,讓它自己解壓。過程中又踩到:

- 安裝程式靠磁片的 `DISK.ID` 驗證「現在插的是第幾片」,把三片檔案混在一個資料夾會驗證失敗、只解出第一片。得用 DOSBox 的 `imgmount` 掛三個磁片映像,在換片提示時送 Ctrl+F4 切換映像。
- 全程無人值守,要用虛擬顯示(Xvfb)+ 模擬按鍵(xdotool)自動過安裝流程。

**「取得原始素材」本身就是一個子專案**,這在有現成資料檔的中文化裡是不會遇到的。

---

## 5. floppy 與 CD 對不起來,但腳本結構一樣

融合要把「floppy 的文字」對上「CD 的語音編號(speechId)」。麻煩的是:

- 兩版的字串 **ID 編號不一樣**(floppy 第 0 條是 "Pink splodge",CD 是 "Witch's Cottage"),不能直接用 ID 對映。
- 但兩版的 **子程式索引表(TBLLIST)完全相同** → 遊戲腳本結構一致 → 對白出現的**順序**一樣。

於是對齊方法是:**用 ScummVM 引擎自己的反組譯器**把兩版腳本都 dump 出來,在每個「顯示文字」的 opcode 上,按位置配對「floppy 的文字」與「CD 的 speechId」。實測 1655 個共同子程式幾乎完美對位,證實 CD 的對白 opcode 幾乎全是 `NULL_STRING`(無字幕)——再次印證第 3 點。

**重點:別自己重寫 bytecode parser,把引擎當現成的反組譯工具用。**

---

## 6. 低解析度畫布容不下正常大小的中文

原版是 320×200。中文筆畫多,硬塞進原版那種約 8px 的小字位會糊成一團(前一版就是塞 12×12,幾乎不可讀)。retro 中文化的正解是**用正常字級(24×24)**,而不是把字縮小。

但這又撞到 AGOS 的實作細節:

- **字幕**是畫進一塊很小的 sprite 緩衝(原設計只有 6400 bytes = 2 行 × 320 × 10px 的英文量)。24px 中文多行會直接撐爆,得把這塊緩衝加大。
- **視窗文字**(動詞列、存檔)畫在只有 8px 高的細長條上,24px 會超出——這塊得另外處理版面。

換句話說,「字要多大」不是美術偏好問題,是牽動記憶體配置與版面的工程問題。

---

## 7. 一堆瑣碎但會讓你卡住的細節

- **Big5 換行**:中文沒有空格,原版「找空格斷行」的邏輯完全不適用,得改成按字寬計算。
- **雙位元組狀態機**:繪字函式一次只給一個 byte,得暫存 Big5 首 byte,收到第二個 byte 才組合繪製。
- **檔名大小寫**:DOS 原版檔名全大寫(`GAMEPC`、`0012.VGA`),在區分大小寫的 Linux 上,不同載入路徑對大小寫的容忍度不一,弄錯就「找不到檔」。
- **回傳緩衝上限**:引擎原本的字串緩衝只有 180 bytes,長中文對白(Big5 每字 2 bytes)會被截斷,得改用更大的緩衝。

這些單獨看都不難,但每一個都足以讓畫面變黑或崩潰,而且症狀往往長得一樣(「沒畫面」),得一個一個排除。

---

## 8. 給後來者的建議

1. **先查範圍再翻譯**:把「每一種螢幕文字從哪讀、怎麼畫」列成表(見 [`strings/scope.md`](../strings/scope.md)),再動手。分母沒搞清楚就翻,一定漏。
2. **別信自我標記的「完成」**:以「實機看得到中文 + dump 未翻字串歸零」為驗收,不以翻譯表條數自稱完成。
3. **把引擎當工具**:反組譯、解壓、載入格式,能讓引擎自己做就別重寫。
4. **完整性優先**:老遊戲是正在消失的數位文物,難的、醜的、少人看的部分越要保全,不用投報率當放棄的藉口。
